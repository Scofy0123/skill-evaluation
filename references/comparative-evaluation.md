# 多候选 Skill 对比评测

当用户要求比较两个或多个 skill / agent / 工作流哪个更好用时，使用本文件。目标不是把多个评测报告拼在一起，而是在同一任务、同一数据、同一评判口径下形成可解释的对照结论。

## 1. 启用条件

适用于：

- 两个或多个 skill 解决相近任务，例如两个 PRD 生成 skill。
- 同一 skill 的多个版本、分支或 prompt 方案对比。
- 一个候选 skill 与无 skill 基线、旧流程、人工流程或竞品流程对比。

不适用于职责明显不同的对象强行排名。职责不同但用户仍要比较时，先把比较目标改写为“场景适配性对比”，不要给全局胜负。

## 2. 评测对象固定

在确认包和报告中先列候选清单：

| 字段 | 说明 |
| --- | --- |
| 候选 ID | 稳定短名，如 `skill_a`、`skill_b`、`baseline` |
| 名称 | skill 名称或仓库路径 |
| 来源 | 本地路径、GitHub URL、飞书文档或其他来源 |
| 版本证据 | commit、文件 SHA、读取时间、文档版本或快照路径 |
| 主功能产物 | 该候选最核心应产出什么 |
| 适用边界 | 该候选声明负责什么、不负责什么 |

对比评测必须固定每个候选的证据版本。历史缓存只能作线索，不能替代当前来源；如果只能用历史缓存，候选来源标为 `历史缓存/间接证据`。

## 3. 共同数据集

共同数据集要同时服务“候选间比较”和“存在必要性”。设计顺序：

1. 定义共同任务口径：所有候选都应该完成的核心任务。
2. 定义共同场景体系：先确定 `场景体系名称` 和 `主分类维度`，再列同维度、同粒度的核心测试场景。
3. 标注覆盖标签：语言/地域、技术主题、行业、风险点等作为横切标签，不和核心场景并列。
4. 在每个核心场景内部构建 `优化集 / 保留集`，不能把不同场景分属不同集合。
5. 每条样本标注来源：真实样本、用户提供、历史产物、合成样本或开源样例。
6. 建立候选证据矩阵：`样本 × 候选 skill × 无 skill 基线`，先复用已验证有效的证据单元，再补跑缺失或失效单元。

共同场景体系示例：

| 字段 | 说明 |
| --- | --- |
| 场景体系名称 | 本轮对比共用的场景体系 |
| 主分类维度 | 核心场景采用的唯一主维度 |
| 核心场景 ID | 稳定编号 |
| 核心场景名称 | 同一主维度下的核心场景 |
| 场景定义 | 样本进入该场景的判定标准 |
| 预期观察点 | 该场景希望拉开的候选差异 |
| 覆盖标签 | 横切标签，如技术主题、语言/地域、行业、风险点 |

场景体系示例、反例和 PRD 生成类场景拆分参考 [scenario-design-examples.md](scenario-design-examples.md)；对比评测正文只要求所有候选使用同一套场景体系。

推荐文件：

| 文件 | 内容 |
| --- | --- |
| `候选Skill清单.csv` | 候选 ID、来源、版本、主功能产物、边界 |
| `主功能样本记录.csv` | 样本 ID、场景体系名称、主分类维度、核心场景、覆盖标签、场景内拆分、输入、预期观察点、来源 |
| `可唤起性样本记录.csv` | 路由样本、预期触发候选、负例说明 |
| `场景级拆分矩阵.csv` | 每个核心场景内优化集 / 保留集数量和比例 |
| `标签覆盖矩阵.csv` | 每个覆盖标签的样本数量、覆盖目的和缺口 |
| `候选运行矩阵.csv` | 样本 ID、候选 ID、运行状态、输出路径、日志路径、模型及版本、评判规则、是否复用、复用来源运行 ID、fallback 原因 |

## 4. 独立运行与兜底规则

候选间比较必须尽量隔离上下文：

- 每个候选的同一条样本使用同一输入和同一输出要求。
- 独立运行时保存 JSONL、stderr、退出码、prompt 和产物路径。
- 无 skill 基线使用同一输入，但不加载候选 skill 的内容。
- 先检查候选证据矩阵，已验证有效的候选结果可以复用；只补跑缺失、失效或版本不一致的候选结果和无 skill 基线。
- 失败样本最多按确认方案重试；仍失败则记录失败，不用当前会话补写后计入质量结论。
- 兜底产物可以作为排障材料，但必须标记为 `兜底/非独立证据`，不得进入候选质量分或胜负判断。
- 每次兜底都必须记录原因分类：网络问题、权限/凭证问题、外部服务限流或超时、运行环境/依赖问题、skill 逻辑问题、输入数据问题、评判器问题、用户中断、未知原因。

## 5. 资源效率与结果复用

对比评测通常成本较高，必须先复用有效证据，再补跑缺口。

可复用条件：

| 条件 | 要求 |
| --- | --- |
| 样本一致 | 样本 ID、输入版本和输出要求一致 |
| 候选一致 | 候选来源、版本证据、skill 文件 SHA 或提交信息一致 |
| 模型可追溯 | 任务执行模型及版本或可观测模型标识已记录 |
| 产物可追溯 | 输出路径、日志路径、JSONL / stderr 或等价 trace 可访问 |
| 非兜底 | 不是 fallback 产物，也不是当前会话手写补产物 |

记录字段：

| 字段 | 说明 |
| --- | --- |
| 是否复用 | 是 / 否 |
| 复用来源运行 ID | 被复用结果的运行 ID |
| 复用产物路径 | 输出、日志或 trace 路径 |
| 复用原因 | 为什么该结果仍然有效 |
| 是否计入质量结论 | 是 / 否；不满足可复用条件时写否 |

## 6. 三层对比观察点

| 层级 | 对比重点 | 常见证据 |
| --- | --- | --- |
| 运行框架层 | 候选可唤起性、运行成功率、日志完整性、错误可归因、可移植性 | metadata、router 样本、Codex JSONL、stderr、文件 SHA |
| 能力链路层 | 澄清流程、模板 / Schema 完整性、工具编排、trace 可解释性、中间产物保存 | SKILL.md、references、运行产物、trace 图或 trace 表 |
| 业务效果层 | 同题输出质量、业务可读性、研发可执行性、风险覆盖、相对基线收益 | 产物成对比较、规则/模型/人工评判、基线对照 |

评判方式字段必须沿用通用枚举：`规则评判`、`模型评判`、`人工评判`。脚本、Schema、文件、退出码、正则和阈值检查都写成 `规则评判`；不要在对比评测里另写其他评判方式。

## 7. 指标与胜负口径

不要默认合成单一总分。优先输出以下表：

| 表 | 用途 |
| --- | --- |
| 候选维度胜负表 | 展示每个维度哪个候选更好、证据是什么 |
| 场景适配表 | 展示哪个候选适合哪个核心场景；归因必须基于同一场景体系和覆盖标签 |
| 样本级成对比较表 | 展示同一样本下各候选输出的差异 |
| 存在必要性比较表 | 展示候选 skill 是否超过无 skill 基线 |
| 失败与中断表 | 展示失败是否来自候选能力、环境、权限或外部服务 |

所有分数型对比指标都必须记录评分细则路径或说明。模型评判必须记录模型提示词文件和大语言模型及版本；人工评判必须记录人工评审步骤和评分档位。

候选间胜负标签建议使用：

| 标签 | 含义 |
| --- | --- |
| `候选 A 胜出` | A 在该维度有直接证据优于其他候选 |
| `候选 B 胜出` | B 在该维度有直接证据优于其他候选 |
| `并列` | 候选表现接近，差异不足以支持胜负 |
| `场景相关` | 不同场景下胜负不同 |
| `证据不足` | 样本量、运行覆盖或评判方式不足 |

## 8. Trace 表达

每个候选至少记录三类 trace：

| Trace | 内容 |
| --- | --- |
| 静态 trace | SKILL.md、references、脚本、声明的流程和模板 |
| 运行 trace | 独立运行 JSONL、工具调用、stderr、退出码 |
| 产物 trace | 输入、输出、评判结果、保存路径 |

当图能帮助读者理解差异时，给每个候选画一张流程或 trace 图。图应解释“这个候选如何从输入走到产物”，不要只画装饰性流程。

## 9. 报告必须回答的问题

对比报告的执行摘要必须直接回答：

1. 当前证据下哪个候选更好用。
2. 哪些场景适合候选 A，哪些场景适合候选 B。
3. 候选之间主要差距是什么。
4. 是否都优于无 skill 基线。
5. 结论是否完整；如果是阶段性结论，缺哪些场景或样本。

如果运行被用户暂停或覆盖不足，报告标题和摘要要写成阶段性评估，不要伪装成完整最终结论。

## 10. 产物清单

多候选对比评测推荐至少输出：

- `对比评估报告.md`
- `候选Skill清单.csv`
- `主功能样本记录.csv`
- `可唤起性样本记录.csv`
- `场景级拆分矩阵.csv`
- `标签覆盖矩阵.csv`
- `候选运行矩阵.csv`
- `资源复用记录.csv`
- `指标结果记录.csv`
- `存在必要性成对比较.csv`
- `候选维度胜负表.csv`
- `候选Trace记录.csv`
- `人工评测填写表.csv`
- `证据来源记录.md`
- `生成产物/`
- `无Skill基线对照/`
- `运行日志/`
- `运行错误日志/`
- 必要时保存候选 trace 图，如 `图1-评测总流程.mmd`、`图2-候选A-trace.mmd`
