# 指标与评判方式

定义可测量评估标准时，使用这份说明。可唤起性和存在必要性参考 skill 评测中“是否能正确调用 skill”和“有无 skill 对照组”的思路；模型评判输出结构参考 evaluator 的 key / score / comment 形式。[1][5]

## Skill 专项目标

### 可唤起性

这一项回答：agent 在需要时能不能找到并选中这个 skill。[1]

可测指标：

| 指标 | 计算规则 / 判断口径 |
| --- | --- |
| 目标提示词命中率 | 应触发且实际触发的提示词数 / 全部正例提示词数 |
| 负例不误触率 | 不应触发且实际未触发的提示词数 / 全部负例提示词数 |
| 相似 skill 干扰正确率 | 存在相似 skill 时，正确选择目标 skill 或目标链路的次数 / 全部干扰提示词数 |
| 描述清晰度 | 人工或模型判断 `name` 和 `description` 是否暴露了正确使用场景 |

提示词类型：

- 直说型：明确要求使用该 skill
- 隐含型：只描述任务，不说 skill 名称
- 相似 skill 干扰型：多个相近 skill 都可能匹配
- 负例型：出现相似词，但任务不在本 skill 范围内

### 存在必要性

这一项回答：为什么要把能力沉淀成 skill，而不是依赖通用模型临场处理。[1]

将目标 skill 与无 skill 基线对比：

| 维度 | 对比内容 |
| --- | --- |
| 质量 | 任务正确性、相关性、完整性 |
| 稳定性 | 多次运行结果波动 |
| 可复现性 | 是否能基于产物复现输出 |
| 效率 | 已采集到的耗时、资源使用、人工评判负担 |
| 资产沉淀 | 数据集、评判口径、可复测样本、日志 |

只有独立、受控的无 skill 基线或等价隔离对照，才能支撑 `通过` 或 `部分通过`。如果没有跑无 skill 基线，标为 `待补测`；如果只有历史产物、静态分析、弱隔离 prompt 或同上下文人工估计，标为 `间接证据 + 待独立基线`，不要写成确定性提升。

资源效率口径：

- 做存在必要性评测时，不要求重复跑已经有效落盘的“有 skill”侧结果。只要样本、输入、候选版本、模型及版本、产物路径和日志路径都可追溯，就可以复用前序主功能评测结果。
- 复用有 skill 侧结果后，只补跑独立或等价隔离的无 skill 基线，再做样本级成对比较。
- 复用必须在运行记录、样本记录和指标结果中写清来源运行 ID、复用产物路径、复用原因和是否计入质量结论。
- fallback 产物、当前会话手写补产物、输入不一致、候选版本不一致、模型版本不可追溯或日志缺失的结果，默认不能作为正式质量结论复用。

## 评判方式选择

`评判方式` 字段只能填写以下枚举值，可多选但不能写其他值：

| 评判方式 | 适用场景 | 评判规则必须记录 |
| --- | --- | --- |
| 规则评判 | 命令退出码、文件生成、Schema、正则、阈值、脚本、确定性业务规则、路由日志、同题成对比较表 | 具体规则、阈值、字段清单、脚本文件路径或数据表路径 |
| 模型评判 | 语义相关性、内容完整性、业务可读性、风险识别、模糊质量初筛 | 模型提示词文件路径、使用的大语言模型及版本、评分细则 |
| 人工评判 | 业务满意度、团队口径、金标争议、上线阈值、最终质量裁决 | 人工评审步骤、通过标准、评分档位、需填写字段、评审人责任 |

脚本、Schema 校验、文件存在性、命令退出码、正则和阈值检查都属于 `规则评判`。不要把它们写成其他评判方式。

## 评判规则字段写法

| 评判方式 | 写法 |
| --- | --- |
| 规则评判 | 写清可复现规则，例如“输出文件存在且非空”“必须包含 X/Y/Z 字段”“脚本 scripts/check_schema.js 返回 0”。使用脚本时必须记录脚本路径或链接。 |
| 模型评判 | 写清模型提示词文件路径，例如 `judging/prompts/prd_quality_judge.md`，并记录模型名称、版本或可观测标识。 |
| 人工评判 | 写清人工如何看、按什么标准给结论、哪些字段必填、何时需要二次确认。 |

## 评分细则

任何分数型指标都必须配套评分细则，不能只给分数。评分细则至少包含：

| 字段 | 说明 |
| --- | --- |
| 分值档位 | 例如 0 / 1 / 2 / 3 或 1-5 分 |
| 每档判定条件 | 每个分值对应的可观察证据 |
| 证据要求 | 需要引用的产物路径、日志、字段或样本文本 |
| 扣分规则 | 缺字段、错误、过度编造、格式不合规等如何扣分 |
| 人工复核触发条件 | 低置信度、高风险、模型/规则分歧或临界分数时何时交给人工 |

模型评分器的 prompt 文件必须包含同一套评分细则，报告和数据记录表都要引用该 prompt 文件路径。

模型评判输出应结构化。建议至少包含：

| 字段 | 含义 |
| --- | --- |
| `key` | 指标名或判断维度 |
| `score` | 数值分或分类标签 |
| `comment` | 简洁理由，包含证据 |
| `confidence` | 可选置信度标签 |
| `needs_human_review` | 模糊或高影响样本标为 true |

这个结构参考 LangSmith 等 evaluator 框架中可聚合的 key / score，以及便于人工复核的解释性 comment。[5]

## 评测场景体系

设计测试数据前，先定义场景体系。场景体系用于回答“本轮为什么选这些场景”，避免把不同维度的样本混成一组。

场景体系必须满足：

- 只有一个主分类维度，例如输入成熟度、任务阶段、用户角色、风险类型、输出形态、链路步骤。
- 核心测试场景来自同一主分类维度和同一粒度。
- 语言/地域、技术主题、行业、风险点、AI/RAG 等横切因素写入覆盖标签。
- 标签覆盖矩阵只能说明横切覆盖情况，不能替代核心场景和场景内优化集 / 保留集拆分。

推荐记录字段：

| 字段 | 说明 |
| --- | --- |
| 场景体系名称 | 本轮场景体系的稳定名称 |
| 主分类维度 | 核心场景采用的唯一主维度 |
| 核心场景 ID | 稳定场景编号 |
| 核心场景名称 | 同一主维度下的核心场景 |
| 场景定义 | 样本进入该场景的判断标准 |
| 预期观察点 | 这个场景希望观察的能力差异 |
| 覆盖标签 | 横切标签，如技术主题=AI/RAG、语言/地域=中文业务 |
| 优化集样本数 | 该核心场景内部进入优化集的样本数量 |
| 保留集样本数 | 该核心场景内部进入保留集的样本数量 |
| 是否场景级合规 | 是否同维度、同粒度，并完成场景内拆分 |

具体场景体系示例、反例和常见误区参考 [scenario-design-examples.md](scenario-design-examples.md)。

## 分歧分析

当不同评判方式不一致时，建立分歧样本表。

推荐字段：

| 字段 | 说明 |
| --- | --- |
| 样本 ID | 稳定 ID |
| 场景体系名称 | 本轮场景体系的稳定名称 |
| 主分类维度 | 核心场景采用的唯一主维度 |
| 核心测试场景 | 样本所属的核心场景 |
| 覆盖标签 | 横切标签，如语言/地域、技术主题、行业、风险点 |
| 数据集 | 该测试场景内部的优化集 / 保留集；未形成场景级拆分时写临时观察样本 |
| 样本类型 | 直接相关、边界样本、强排除等 |
| 规则结果 | 输出媒介支持时使用醒目标识 |
| 模型结果 | 尽量使用醒目标识 |
| 分歧方向 | 规则低估 / 规则高估 / 边界不清 |
| 标题或输入 | 简洁原始输入 |
| 分歧理由 | 为什么发生分歧 |
| 人工结果 | 未复核时留空 |
| 人工备注 | 未复核时留空 |

解释口径：

- `规则低估`：规则判无关或低分，语义评判认为相关。
- `规则高估`：规则判相关或高分，语义评判认为无关或较弱。
- `边界不清`：两种判断都有理由，需要业务策略定夺。

## 结果呈现

有实测结果时优先展示实测值。未测项使用以下标签：

- `已测`：本轮已经测得
- `间接证据`：有产物支持，但不是受控测试
- `待补测`：尚未测量，且已经判断当前工具、权限、数据或目标对象形态无法主动补测
- `人工待复核`：需要人工金标判断

不要把不确定性藏在长段文字里。

报告首页需要把运行框架层和能力链路层的关键检查项做成状态总览。状态建议使用：

| 状态 | 使用场景 |
| --- | --- |
| `✅` | 本轮有直接证据证明满足 |
| `❌` | 本轮有直接证据证明不满足 |
| `部分满足` | 关键路径可用，但有边界、稳定性或覆盖不足 |
| `待补测` | 尚无直接证据，且当前无法通过脚本、规则评判、历史产物复盘、独立会话基线或可用 API 主动补测 |

状态总览至少覆盖：

- 可唤起性 / 路由选择
- 运行成功率
- 已采集事件、执行日志、错误可观测性
- 中间产物完整性
- 工具调用和状态交接
- 评判结果可复现性

多候选对比评测的结果呈现要额外覆盖：

| 表 | 字段 |
| --- | --- |
| 候选对象表 | 候选 ID、来源、版本证据、主功能产物、适用边界 |
| 评测场景体系表 | 场景体系名称、主分类维度、核心场景、场景定义、预期观察点、覆盖标签 |
| 候选运行矩阵 | 样本 ID、核心测试场景、覆盖标签、候选 ID、运行状态、输出路径、日志路径、是否计入质量结论 |
| 候选维度胜负表 | 维度、胜出候选、证据、适用核心场景、限制 |
| 场景适配表 | 核心测试场景、覆盖标签、推荐候选、推荐理由、反例或风险 |
| 样本级成对比较表 | 样本 ID、候选 A 结果、候选 B 结果、无 skill 基线结果、胜负标签、解释 |
| 候选 trace 表 | 候选 ID、静态 trace、运行 trace、产物 trace、错误 trace |

胜负标签建议使用 `候选 A 胜出`、`候选 B 胜出`、`并列`、`场景相关`、`证据不足`。如果运行覆盖不足，只能写阶段性结论，不能把局部样本外推为全局胜负。

## Codex 事件日志

如果评测通过 Codex / Codex Desktop / Codex CLI 执行，优先使用 Codex 本地事件作为运行框架层证据。不要笼统写“缺少 trace”；先检查是否能拿到以下事件。

| 事件或来源 | 可用信息 | 推荐用途 |
| --- | --- | --- |
| `session_meta` | 会话 ID、cwd、originator、模型、CLI 版本 | 运行环境、复现定位 |
| `turn_context` | turn ID、sandbox、approval、模型设置 | 权限与执行边界 |
| `response_item.function_call` | 工具名、参数、call_id | 工具调用和状态交接 |
| `response_item.function_call_output` | 工具输出、错误、退出状态 | 运行成功率、失败归因 |
| `event_msg.token_count` | input/output/cached/reasoning token usage，如存在 | token usage 记录；不等同于成本金额 |
| `event_msg.task_complete` | 完成时间、耗时、最后消息 | 任务闭环证据 |
| `~/.codex/sessions/.../rollout-*.jsonl` | 交互式会话事件流 | 历史复盘 |
| `codex exec --json` | 非交互 JSONL 事件流 | 自动化评测首选日志 |
| `~/.codex/logs_2.sqlite` | Codex 内部日志、SSE / OTel 风格事件 | 辅助排查 |

报告只写本轮实际采集到的观测项。`token 成本`、`总成本`、`在线写入耗时`、`真实 trace UI` 这类指标没有直接采集到时，不要放进状态总览或指标表，也不要写成“缺少”。可以改写为“已保存 Codex session JSONL / CLI JSONL / 工具调用日志”。

注意：上面这套 Codex 事件说明是评测者的采集口径，不是最终报告正文。写报告时必须转成事实表述，例如“本轮采集到 session JSONL：<path>”“可复盘工具调用和输出”“未执行真实写入，因此该链路未评估”。不要把“如果评测通过 Codex...优先使用...”这类指导语原样贴进报告。

## 数据记录表

评判输出必须保存到可复用的数据记录表。前期可以是 CSV / JSONL / Markdown 表，后续可以升级为 SQLite、LangSmith、飞书多维表格或正式数据库。

推荐拆成三类记录：

| 表 | 粒度 | 典型字段 |
| --- | --- | --- |
| 评测运行记录 | 每轮评测运行的整体记录 | 运行 ID、候选 ID 或 variant、命令、环境、数据版本、任务执行模型及版本、模型标识来源、开始时间、结束时间、已采集事件时间、Codex 事件日志路径、输出目录、是否补跑、复用来源运行 ID |
| 测试样本记录 | 每条样本或一个任务的输入输出记录 | 运行 ID、样本 ID、候选 ID 或 variant、场景体系名称、主分类维度、核心测试场景、覆盖标签、场景内数据集拆分、正式拆分状态、输入、预期输出、实际输出、产物路径、错误、fallback 原因、fallback 是否计入结论、是否复用、复用产物路径 |
| 指标结果记录 | 每个指标的评判结果与解释 | 运行 ID、样本 ID、候选 ID 或 variant、所属层级、指标名、评判方式、评判规则、模型及版本、评分细则路径/说明、分数或标签、通过状态、解释、复核状态、复用证据说明 |

写入 CSV、Sheet 或飞书表格时，指标结果字段不要使用“命中数/总数”这类裸分数。表格工具可能把它们自动识别成日期。应写成 `4条中4条（100%）`、`3条中3条（100%）` 这类稳定文本，只有真实日期字段才写日期。

如果只维护一个表，至少包含：

| 字段 | 说明 |
| --- | --- |
| 运行 ID | 本轮评测运行的稳定 ID |
| 样本 ID | 测试样本的稳定 ID |
| 候选 ID / variant | 单 skill 评测可为空；多候选对比时写候选 skill、版本或无 skill 基线 |
| 场景体系名称 | 本轮场景体系的稳定名称 |
| 主分类维度 | 核心场景采用的唯一主维度 |
| 核心测试场景 | 样本所属的核心测试场景 |
| 覆盖标签 | 横切标签，如技术主题、语言/地域、行业、风险点 |
| 场景内数据集拆分 | 使用该场景内部已确认的数据集类别；未纳入正式拆分时写 `临时观察样本（未纳入正式拆分）` |
| 正式拆分状态 | 场景级合规 / 场景级覆盖不足 / 临时观察样本 |
| 输入 | 输入文本或文件路径 |
| 预期输出 | reference output 或人工金标路径 |
| 实际输出 | 实际输出路径 |
| 所属层级 | 三层之一 |
| 指标名 | 被评判的指标 |
| 评判方式 | 只能填写 `规则评判`、`模型评判`、`人工评判`，可多选 |
| 评判规则 | 具体规则、脚本路径、模型提示词路径或人工评审步骤 |
| 大语言模型及版本 | 任务执行模型、模型评判模型、无 skill 基线模型；没有用到时写无 |
| 评分细则路径/说明 | 分数型指标必填；非分数型指标写不适用 |
| 分数或标签 | 数值分或分类标签 |
| 通过状态 | 通过 / 不通过 / 部分通过 / 待补测 |
| 解释 | 简洁解释 |
| 产物路径 | 日志、trace、截图或输出文件路径 |
| 复核状态 | 未复核 / 已复核 / 需要人工复核 |
| 是否复用 | 是 / 否；复用前序有效证据时写是 |
| 复用来源运行 ID | 被复用结果的运行 ID；没有复用时写不适用 |
| 复用产物路径 | 被复用的输出、日志或 trace 路径 |
| 复用原因 | 例如同样本、同输入、同版本、同模型，避免重复跑有 skill 侧 |
| fallback 原因 | 网络问题 / 权限/凭证问题 / 外部服务限流或超时 / 运行环境/依赖问题 / skill 逻辑问题 / 输入数据问题 / 评判器问题 / 用户中断 / 未知原因 |
| fallback 是否计入结论 | 默认否；只有明确标记为非独立证据时才可作为限制性参考 |

多候选对比评测建议额外保存 `评测场景体系表.csv`、`标签覆盖矩阵.csv`、`候选运行矩阵.csv`、`候选维度胜负表.csv`、`场景适配建议.csv`、`候选Trace记录.csv`。这些表不要只放进报告正文，也要保存成可复用文件。

## 静态评测结果标签

对于没有可执行脚本的文档型 / 工作流型 skill：

| 项目 | 标记 |
| --- | --- |
| frontmatter 校验 | `已测` |
| reference / template 覆盖 | `已测` |
| 安全边界检查 | `已测` |
| 没有 router 日志的 skill 唤起行为 | 先做静态提示词 / 相似 skill 干扰仿真，标 `已测` 或 `间接证据`；真实 router trace 无法取得时，该 trace 单项标 `待补测` |
| 没有真实任务运行的最终输出质量 | 先尝试历史输出复盘、规则评判或构造样本；仍无输入/输出证据时才标 `待补测` |
| 没有无 skill 基线的存在必要性 | 如已有有效有 skill 结果，复用该侧结果并只补跑无 skill 基线；仍无法运行时才标 `间接证据` + `待补测` |

## Metadata 兼容性

校验目标 skill 时，要区分“不可用”和“可移植性风险”：

| 发现 | 报告方式 |
| --- | --- |
| 严格 validator 因 `name` 含下划线或大写而失败 | 除非当前 runtime 明确拒绝，否则标为 `可移植性风险` |
| 缺少 `description` | 标为 `阻塞问题`，因为 skill 唤起依赖它 |
| description 较宽泛但基本合理 | 标为唤起风险，并补相似 skill 测试 |
| 命令里有本地绝对路径 | 标为可移植性风险，并建议参数化或补安装说明 |

不要只因为某个 validator 的命名规则更严格，就断言该 skill 在当前环境不可用。
