# Skill Evaluation Runbook

本文件是 `skill-evaluation` 的运行参考手册，供 agent 在执行评测时按章节读取。它不是面向读者的飞书工作法正文，也不要求和飞书《Skill / Agent 评测流程（新版）》逐字同步。

使用方式：

- `SKILL.md` 负责给出硬性原则和执行入口。
- 本文件负责把评测执行拆成稳定的 `RUN-xx` 步骤。
- 具体指标、报告模板、飞书写入和对比评测细则，按需读取同目录下的专项 reference。

## RUN-00 适用范围与总原则

适用于单个 skill、多个候选 skill 对比、多环节 skill、接近 agent 的工作流，以及基于历史产物或日志的复盘评测。

核心原则：

| 原则 | 执行要求 |
| --- | --- |
| 先理解对象再设计评测 | 先阅读目标 `SKILL.md`、相关代码、引用文件、日志、历史产物和用户需求 |
| 固定三层观察 | 始终使用 `运行框架层 / 能力链路层 / 业务效果层` |
| 必须评估可唤起性 | 检查 name、description、触发提示词、负例和相似 skill 干扰 |
| 必须评估存在必要性 | 与无 skill 基线或等价隔离对照比较，证明 skill 相对通用模型临场处理的收益 |
| 场景体系化 | 核心场景必须来自同一主分类维度，语言、行业、技术主题等写成覆盖标签 |
| 场景内拆分数据集 | 每个核心场景内部都要拆 `优化集 / 保留集`；不能用不同场景分属不同集合替代拆分 |
| 评判方式枚举 | `评判方式` 只能使用 `规则评判 / 模型评判 / 人工评判` |
| 模型和评分可追溯 | 记录任务执行模型、模型评判模型、无 skill 基线模型；分数型指标必须有评分细则 |
| fallback 可归因 | 记录原因分类、触发环节、处理动作和是否计入质量结论 |
| 节约 token 与运行资源 | 优先复用已验证有效的运行结果，只补跑缺失或失效的证据单元 |
| 优化后补样复测 | 修改 skill 后，把目标改善点转成补充样本或标签，重跑受影响优化集并抽测保留集 |
| 关键节点要澄清 | 目标/指标、三层拆解、数据来源、数据集构建、人工评判、权限副作用和输出形态不能由评测者自行猜定 |

资源效率原则：

- 评测前先盘点已有有效证据，避免盲目重跑。
- 做存在必要性评测时，如果同一批样本已产生有效“有 skill”产物，可以复用为对照侧证据，只补跑“无 skill”基线。
- 多候选对比时，先建立证据矩阵，再补齐缺失格子；不要求无条件重跑完整 `样本 × 候选 × 基线`。
- 复用结果必须记录来源运行 ID、样本 ID、候选 ID、模型及版本、输入版本、产物路径、日志路径、复用原因和是否计入正式结论。
- 以下结果默认不能作为正式质量结论复用：fallback 产物、当前会话手写补产物、输入不一致、skill 版本不一致、模型版本不可追溯、日志或产物路径缺失。

优化后补样复测：

- 当修改 skill 的 description、规则、模板、工具链、评分器或参考文件时，先写清本次修改希望改善的具体问题。
- 将该问题转成新增样本、覆盖标签或人工评判项；新增样本优先进入对应场景的优化集，不能替代原有保留集。
- 复测时只重跑受影响的证据单元，同时抽测相关保留集和边界样本，确认预期改善成立且没有破坏既有能力。
- 记录修改点、补充样本 ID、重跑范围、对照结果、是否符合预期和仍需补测的缺口。

## RUN-01 收集上下文

先收集能支持评测设计的证据，不直接写结论。

需要读取：

- 用户需求、业务场景、成功标准和输出要求。
- 目标 `SKILL.md`、`references/`、`scripts/`、`assets/`、README、agents metadata。
- 目标实际调用的本地代码路径、CLI、API、配置和依赖。
- 已有日志、trace、metrics、数据集、历史报告和产物。
- 如果通过 Codex 执行，优先采集 session JSONL、`codex exec --json`、工具调用输出和可用 token_count。

记录上下文时区分：

| 类型 | 记录方式 |
| --- | --- |
| 当前在线或本地真实来源 | 可作为直接证据 |
| 历史缓存或旧报告 | 只能作为线索，除非报告中标为间接证据 |
| 缺权限或无法读取 | 记录阻塞原因和下一步补齐方式 |

如果用户没有给出数据集、历史样本、金标或参考产物，先记录为 `待用户确认`。不要在此步骤把“评测执行者构造合成样本”默认为正式数据来源。

## RUN-02 理解评测对象

目标是回答：这个 skill / agent 到底负责什么，产出什么，边界在哪里。

基础字段：

| 字段 | 记录内容 |
| --- | --- |
| 使用场景 | 谁在什么业务场景中使用 |
| 成功标准 | 什么结果算有用，哪些判断需要人工确认 |
| 核心任务 | 目标对象主要负责什么 |
| 主功能产物 | PRD、测试用例、代码补丁、报告、表格、飞书文档等 |
| 输入 | prompt、文件、数据集、API、上游产物 |
| 输出 | 文件、结构化结果、判断、动作或在线文档 |
| 依赖 | API、key、CLI、模型、仓库、外部工具 |
| 边界 | 不负责什么，哪些副作用需要授权 |

多候选对比额外记录：

| 字段 | 记录内容 |
| --- | --- |
| 候选 ID | 每个候选的稳定短名 |
| 候选来源 | 本地路径、GitHub URL、飞书文档或其他来源 |
| 版本证据 | commit、SHA、读取时间、快照路径或文档版本 |
| 共同任务口径 | 所有候选都应完成的核心任务 |
| 候选差异假设 | 预期在哪些场景或维度拉开差距 |
| 无 skill 基线 | 是否纳入、如何隔离运行 |

需要向用户澄清的问题：

| 澄清点 | 何时必须问 |
| --- | --- |
| 业务目标和成功标准 | 用户只说“评测一下”“看看哪个好”，但没有说明业务上什么算好 |
| 主功能产物 | skill 声明宽泛，或实际可能产出多种文件/动作 |
| 候选边界 | 多候选职责不完全一致，可能不适合直接排名 |
| 人工判断口径 | 涉及业务满意度、团队口径、上线阈值、视觉审美或金标争议 |
| 数据来源 | 用户没有提供样本、历史产物、真实数据或允许构造合成样本 |

## RUN-03 建立三层模型

三层用于定位问题来源，不按代码模块切分。

| 层级 | 观察点 | 常见证据 |
| --- | --- | --- |
| 运行框架层 | 唤起、入口、权限、凭证、依赖、日志、trace、错误、重试、运行状态 | metadata、router 样本、命令、退出码、Codex JSONL、stderr |
| 能力链路层 | 输入拆解、数据转换、检索、解析、生成、评分、工具编排、中间产物和 Schema | references、脚本、运行产物、Schema 校验、步骤日志 |
| 业务效果层 | 最终质量、相关性、准确性、完整性、可读性、安全边界、误判漏判、人工评判、相对基线收益 | 输出产物、样本级比较、规则/模型/人工评判、无 skill 基线 |

普通单步 skill 只保留必要指标；复杂工作流在三层内部补充观察点，不新增第四层。

三层模型形成后，要在确认包中让用户确认职责边界是否准确。若用户认为某个观察点不属于目标 skill 的责任，先调整三层拆解和指标，再进入正式测试。

## RUN-04 定义目标和指标

每次评测至少覆盖：

1. 运行目标：能否稳定运行。
2. 能力目标：内部链路能否产出可用中间结果。
3. 业务目标：是否解决用户或业务问题。
4. 唤起目标：agent 能否正确选择这个 skill。
5. 必要性目标：相比无 skill 基线是否有质量、稳定性、可复现性或资产沉淀收益。

指标表字段：

| 字段 | 要求 |
| --- | --- |
| 指标名 | 简洁中文名称 |
| 所属层级 | 固定三层之一 |
| 计算规则 | 明确公式或判断口径 |
| 数据来源 | 日志、JSON、trace、模型输出、人工标签 |
| 评判方式 | 只能填写 `规则评判`、`模型评判`、`人工评判`，可多选 |
| 评判规则 | 规则、脚本路径、模型提示词路径或人工评审步骤 |
| 大语言模型及版本 | 任务执行模型、模型评判模型、无 skill 基线模型 |
| 评分细则路径/说明 | 分数型指标必填 |
| 复用证据 | 若使用历史或前序运行结果，记录来源运行 ID、产物路径和复用原因 |
| 结果 | 实测值、待补测、间接证据或人工待评 |

设计指标时提醒用户补充更关心的业务目标、重点场景、风险点和人工判断口径。

如果目标、风险或人工口径不清楚，指标表只能作为草案。不要把评测者自行选择的业务权重、场景优先级或评分口径写成已确认方案。

## RUN-05 确认评测设计

正式测试、在线写入或大规模运行前，必须先输出确认包并暂停，除非用户明确要求跳过确认。

确认包最小结构：

| 顺序 | 模块 | 必须写清 |
| --- | --- | --- |
| 1 | 评测对象与证据范围 | 目标、已读取来源、缺失范围 |
| 2 | 三层拆解 | 三层分别观察什么 |
| 3 | 评测目标与指标表 | 指标、层级、计算规则、数据来源、评判方式、评判规则、模型版本、评分细则、复用证据 |
| 4 | 数据方案与拆分 | 场景体系、主分类维度、核心场景、覆盖标签、优化集/保留集、缺口 |
| 5 | 评判方式与评判规则 | 三类枚举、脚本/提示词/人工规则、模型版本 |
| 6 | 主功能产物评测方案 | 产物如何生成、保存和评判 |
| 7 | 人工评测项与填写表方案 | 哪些内容必须人工评估、字段和保存位置 |
| 8 | 测试流程与数据记录表 | 步骤、运行矩阵、复用策略、落盘路径 |
| 9 | 副作用与权限边界 | 在线写入、付费 API、外部系统修改 |
| 10 | 需要用户澄清或确认的关键问题 | 明确请求确认后再执行 |

确认前可以读取资料、做静态检查和小范围可逆验证；确认前不能批量运行、生成最终报告或写入飞书等在线系统。

关键澄清问题至少覆盖：

- 评测目标和指标是否符合用户真正关心的业务问题。
- 三层拆解是否符合目标 skill / agent 的真实职责边界。
- 用户是否能提供真实数据集、历史样本、生产样例、金标或参考产物。
- 如果用户不能提供数据，是否允许评测者构造合成样本；合成样本是否计入正式结论。
- 样本规模、场景体系、优化集 / 保留集比例是否接受。
- 人工评判项由谁填写，按什么标准判定。
- 是否允许外部 API 调用、在线写入、上传文件或其他副作用。

## RUN-06 设计测试流程

设计顺序：

1. 定义共同任务口径。
2. 设计评测场景体系。
3. 向用户确认能否提供数据集、历史样本、金标或参考产物。
4. 构建样本池并标注来源。
5. 如果需要合成样本，先说明生成原则、样本量、限制和是否进入正式结论。
6. 在每个核心场景内部拆 `优化集 / 保留集`。
7. 选择评判方式和评分细则。
8. 建立运行矩阵和资源复用策略。
9. 定义失败、fallback、重试和排障记录规则。
10. 判断是否需要 multi-agent / sub-agent 隔离观察面。

场景体系字段：

| 字段 | 说明 |
| --- | --- |
| 场景体系名称 | 本轮场景体系的稳定名称 |
| 主分类维度 | 本轮核心场景采用的唯一主维度 |
| 核心场景 ID | 稳定场景编号 |
| 核心场景名称 | 同一维度下的核心场景 |
| 场景定义 | 样本进入该场景的判定标准 |
| 预期观察点 | 该场景主要观察什么能力差异 |
| 覆盖标签 | 语言/地域、技术主题、行业、风险点等横切标签 |
| 优化集样本数 | 该核心场景内部的优化集样本数 |
| 保留集样本数 | 该核心场景内部的保留集样本数 |
| 是否场景级合规 | 是否同维度、同粒度，并完成场景内拆分 |

需要判断场景体系是否混杂维度，或需要查看具体示例/反例时，读取 [scenario-design-examples.md](scenario-design-examples.md)。

数据集拆分规则：

- 只使用 `优化集 / 保留集` 两类。
- 拆分基本单位是核心测试场景，不是全量样本池。
- 每个核心场景原则上都要同时覆盖优化集和保留集。
- 全局比例不能替代场景内比例。
- 样本不足时标为 `场景级覆盖不足` 或 `临时观察样本（未纳入正式拆分）`，并说明补样计划。

数据来源口径：

| 数据来源 | 处理方式 |
| --- | --- |
| 用户提供真实数据 | 作为优先数据源，记录来源、授权范围和脱敏要求 |
| 用户提供历史产物 | 可用于复盘评测，标明是否为间接证据 |
| 开源样例或公开数据 | 记录 URL、读取时间和适用限制 |
| 评测执行者构造的合成样本 | 必须经用户确认，标明生成原则、覆盖场景和是否计入正式结论 |
| 数据不足 | 标为场景级覆盖不足或临时观察样本，并写补样计划 |

资源复用策略：

| 场景 | 推荐动作 |
| --- | --- |
| 已有有效“有 skill”产物，需要做存在必要性 | 复用有 skill 侧，只补跑无 skill 基线 |
| 已有部分候选结果，新增候选或基线 | 只补跑缺失候选或基线 |
| skill、输入、模型或版本变化 | 只重跑受影响样本和候选 |
| 结果来自 fallback 或当前会话补写 | 不计入正式质量结论，只作为排障证据 |

## RUN-07 建立数据记录表

执行前先定义数据表结构和存档位置。

最小三类记录：

| 数据类型 | 典型字段 |
| --- | --- |
| 评测运行记录 | 运行 ID、候选 ID、执行命令、环境、数据版本、任务执行模型及版本、模型标识来源、开始/结束时间、Codex JSONL 路径、输出目录、是否补跑、复用来源运行 ID |
| 测试样本记录 | 运行 ID、样本 ID、候选 ID、场景体系、主分类维度、核心场景、覆盖标签、场景内拆分、输入、预期输出、实际输出、产物路径、错误、fallback 原因、fallback 是否计入结论、是否复用、复用产物路径 |
| 指标结果记录 | 运行 ID、样本 ID、候选 ID、所属层级、指标名、评判方式、评判规则、模型及版本、评分细则、分数或标签、通过状态、解释、复核状态、复用证据说明 |

如存在人工评测项，必须额外保存人工评测填写表。表格可为空，但字段必须完整，并绑定具体样本和产物路径/链接。

## RUN-08 运行或复盘测试

执行顺序：

1. 先补齐证据矩阵，不盲目重跑已验证有效的结果。
2. 安全且已授权时，优先本地实际运行目标 skill。
3. 无现成入口时，先判断能否写小型评测脚本、做规则评判、静态结构校验、历史产物复盘、飞书/API 证据拉取、构造样本或启动独立无 skill 基线。
4. 对主功能产物执行产物存在性检查，报告必须能指向生成产物或历史产物路径。
5. 对 `待补测 / 待验证` 项先做可主动补测性判断；能测就测，不能测才保留状态。
6. 记录命令、退出码、输出文件、日志、样本级结果、fallback 原因、是否补跑、是否复用。

存在必要性执行口径：

- 可将前序同样本、同输入、同版本、同模型且有产物/日志的“有 skill”结果作为正式对照侧。
- 无 skill 基线必须独立或等价隔离运行，不能加载候选 skill 内容。
- 对比表要能追溯到有 skill 结果和无 skill 结果各自的产物与日志。
- 如果只能使用间接证据，结论写 `间接证据 + 待独立基线`，不能写成已验证稳定收益。

## RUN-09 分析、报告与回流

分析顺序：

1. 先展示测量数据，再解释含义。
2. 当规则、模型或人工结果不一致时，列出分歧样本。
3. 人工评测项未填写前标为 `人工待评`，不能写成通过。
4. 失败点同时沉淀为 eval 资产和 skill 优化项。

报告使用 [report-template.md](report-template.md)。写最终报告前做口吻自检：不要把 skill 内部操作规则原样贴进报告正文，要改写为本轮实际做了什么、采集到什么、没有执行什么、因此结论是什么。

闭环沉淀：

| 沉淀内容 | 建议位置 |
| --- | --- |
| 全量样本、优化集、保留集 | `datasets/` |
| 运行记录和指标结果 | `eval_runs/`、`results/` |
| 产物、日志、trace、错误 | `artifacts/` 或本轮输出目录 |
| 无 skill 基线 | baseline artifact |
| 复用证据矩阵 | 候选运行矩阵、存在必要性成对比较表 |
| skill 修复项 | 优化方案、issue 或 PR |

## RUN-10 静态评测模式

当目标没有安全可执行入口，或用户没有授权外部动作时，使用静态评测模式，但仍保留完整报告结构。

静态评测动作：

| 项目 | 做法 | 结果标签 |
| --- | --- | --- |
| metadata 校验 | 校验 frontmatter、name、description | `已测` |
| 文件盘点 | 阅读 `SKILL.md`、references、scripts、README、agents metadata | `已测` |
| 流程覆盖 | 检查输入、输出、边界、流程、质量检查 | `已测` |
| 依赖与安全 | 识别外部工具、凭证、在线写回、破坏性动作 | `已测` |
| 唤起设计 | 设计正例、负例、相似 skill 干扰提示词；能做静态路由仿真就先仿真 | `已测` 或 `间接证据` |
| 真实输出质量 | 优先运行；无法运行时复盘历史输出或构造可评判样本 | `已测`、`间接证据` 或 `待补测` |
| 无 skill 基线 | 能隔离运行则运行；已有有效有 skill 结果时只补跑基线 | `已测`、`间接证据 + 待独立基线` 或 `待补测` |

标记 `待补测` 前，必须说明已经尝试或判断过哪些当前可用补测路径，以及为什么仍无法覆盖。
