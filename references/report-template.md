# 评估报告模板

使用这个结构产出简洁报告。除非用户要求深度附录，否则控制在管理者 7-8 分钟可读完的长度。

可参考成品样例：

- [单一 Skill 评估报告样例：backend-estimation](report-examples/单一Skill评估报告样例-backend-estimation.md)
- [多 Skill 对比评估报告样例：两个开源 PRD](report-examples/多Skill对比评估报告样例-两个开源PRD.md)

## 执行摘要

> 本节先给结论：这个 skill 是否可运行、是否有价值、主要问题是什么、下一步应该怎么改。

如果是多个候选 skill 对比评测，摘要第一段必须直接回答：当前证据下哪个候选更好用、分别适合什么场景、主要差距在哪里、是否优于无 skill 基线。覆盖不足时写成阶段性结论，并说明未覆盖场景。

先放一张状态总览表，让读者在第一页看到运行框架层和能力链路层的关键检查项是否满足。状态使用 `✅`、`❌`、`待补测`、`部分满足`。

| 层级 | 检查项 | 状态 | 证据 |
| --- | --- | --- | --- |
| 运行框架层 | 可唤起性 / 路由选择 | ✅ / ❌ / 待补测 | router 日志、触发提示词测试 |
| 运行框架层 | 运行成功率 | ✅ / ❌ / 待补测 | 命令、退出码、日志 |
| 运行框架层 | 事件日志与错误可观测 | ✅ / ❌ / 待补测 | Codex session JSONL、`codex exec --json`、CLI 日志、工具调用输出 |
| 能力链路层 | 中间产物完整性 | ✅ / ❌ / 待补测 | schema 校验、输出文件 |
| 能力链路层 | 工具调用和状态交接 | ✅ / ❌ / 待补测 | `response_item.function_call`、`response_item.function_call_output`、步骤日志 |
| 能力链路层 | 评判结果可复现 | ✅ / ❌ / 待补测 | 数据记录表、评判器输出 |

包含一张 5-8 行的关键结论表：

| 结论 | 证据 | 对 skill 的含义 |
| --- | --- | --- |
| 链路可以稳定跑通 | 真实运行证据 | 可以进入保留集验收或持续复测 |
| 质量有主要风险 | 分歧样本或失败数据 | 优化特定步骤 |
| 可唤起性待测 / 已测 | 唤起证据 | 优化 name / description 或路由 |
| 存在必要性待测 / 已测 | 无 skill 基线证据 | 证明或修正 skill 价值 |

多候选对比评测增加一张候选对比结论表：

| 问题 | 当前结论 | 证据 | 限制 |
| --- | --- | --- | --- |
| 哪个更好用 | 候选 A / 候选 B / 场景相关 / 证据不足 | 候选维度胜负表、样本级成对比较 | 覆盖场景和样本量 |
| 适合场景 | 场景到候选的推荐 | 场景适配表 | 未覆盖场景 |
| 主要差距 | 结构、质量、可唤起、trace、运行稳定性 | 指标结果记录 | 人工待评项 |
| 是否优于无 skill 基线 | 是 / 否 / 部分 / 证据不足 | 存在必要性成对比较 | 基线隔离程度 |

记录本轮使用的大语言模型及版本：

| 用途 | 模型及版本 | 记录来源 |
| --- | --- | --- |
| 任务执行模型 | 待填 | CLI 参数、Codex JSONL、环境配置或人工记录 |
| 模型评判模型 | 待填 | 评分器配置、模型提示词记录或人工记录 |
| 无 skill 基线模型 | 待填 | CLI 参数、Codex JSONL、环境配置或人工记录 |

记录资源效率与结果复用：

| 项目 | 本轮做法 | 证据 |
| --- | --- | --- |
| 是否复用已有有 skill 结果 | 是 / 否 / 不适用 | 复用来源运行 ID、产物路径、日志路径 |
| 是否补跑无 skill 基线 | 是 / 否 / 不适用 | 基线运行记录、JSONL、输出路径 |
| 是否补跑缺失候选结果 | 是 / 否 / 不适用 | 候选运行矩阵 |
| 复用结果是否计入质量结论 | 是 / 否 | 可复用条件和限制说明 |

## Skill 场景理解与分层

> 本节说明评测前先理解了什么：skill 用在什么场景、成功标准是什么、边界在哪里、按三层如何拆。

包含：

- 使用场景
- 成功标准
- 需要人工确认的业务口径
- 评测对象
- 核心链路
- skill 职责
- 边界
- 三层表

如果是多候选对比评测，补充候选对象表：

| 候选 ID | 名称 | 来源 | 版本证据 | 主功能产物 | 适用边界 |
| --- | --- | --- | --- | --- | --- |

## 评测目标与指标体系

> 本节回答“到底评什么”。每个目标都要落到可观测指标和计算规则。

包含：

- 评测目标
- 指标表，列为：指标 / 所属层级 / 计算规则 / 评判方式 / 评判规则 / 大语言模型及版本 / 评分细则路径或说明 / 本轮结果
- 指标选择原则
- 评测设计确认记录：确认人、确认时间、确认后的调整点

多候选对比指标必须包含候选间胜负关系、场景适配性、样本级成对比较、候选 trace 完整性和无 skill 基线差异。

必备指标：

- 成功率 / 错误 / 已采集事件日志
- Schema 或中间产物完整性
- 任务质量
- 分歧或失败数量
- 可唤起性
- 存在必要性

## 评测设计确认

> 本节说明正式执行前确认过哪些设计，避免指标和流程在执行后才大幅返工。

如果没有取得用户确认，报告不能写成正式评测结果，只能输出“评测设计草案”。用户明确确认后，才能继续执行测试并生成最终报告。

使用表格：

| 确认项 | 草案内容 | 确认结果 | 调整记录 |
| --- | --- | --- | --- |
| 三层拆解 | 运行框架层 / 能力链路层 / 业务效果层 | 已确认 / 待确认 | 待填 |
| 评测目标 | 运行目标、能力目标、业务目标、可唤起性、必要性 | 已确认 / 待确认 | 待填 |
| 指标体系 | 指标、计算规则、数据来源、评判方式、评判规则、模型版本、评分细则 | 已确认 / 待确认 | 待填 |
| 数据方案 | 来源、场景体系、覆盖标签、样本池、拆分、reference output | 已确认 / 待确认 | 待填 |
| 测试流程 | 执行步骤、协同编排、产出路径 | 已确认 / 待确认 | 待填 |
| 数据记录表 | 表结构、存储位置、产物路径 | 已确认 / 待确认 | 待填 |
| 多候选对比口径 | 候选清单、共同任务、运行矩阵、胜负标签 | 已确认 / 待确认 | 待填 |

## 评测流程设计

> 本节说明评测如何形成闭环：从理解对象，到沉淀 eval 资产和 skill 优化方案。

包含：

- 闭环流程
- 沉淀什么、沉淀到哪里
- 失败点如何进入后续 eval
- 测试流程内的 multi-agent / sub-agent 协同编排判断：是否启用、为什么、每个角色负责什么
- 评测数据记录方案：表结构、保存目录、运行 ID 规则；如果通过 Codex 执行，写明采集到的 `session_meta`、`turn_context`、`response_item.function_call`、`response_item.function_call_output`、`event_msg.token_count`、`event_msg.task_complete` 等事件，或对应 session JSONL / `codex exec --json` 文件路径
- 多候选对比时，写明候选运行矩阵、无 skill 基线隔离方式、失败 / 中断 / 兜底产物是否计入质量结论。
- 写明资源效率策略：哪些已有结果被复用，哪些证据单元被补跑，为什么满足复用条件。
- 写明评判方式枚举、评判规则保存位置、评分细则保存位置，以及本轮模型版本记录来源。

## 测试方法与执行流程

> 本节说明这次评测怎么执行：数据从哪里来，如何准备，怎么拆分，什么时候用哪种评判方式。

包含：

- 必要时加入流程图
- 步骤表，列为：步骤 / 具体动作 / 对应层级 / 输出物
- 评测场景体系表，列为：场景体系名称 / 主分类维度 / 核心场景 ID / 核心场景名称 / 场景定义 / 预期观察点 / 覆盖标签
- 数据集拆分依据：先列评测场景体系，再展示每个核心场景内的 `优化集 / 保留集` 拆分
- 场景级拆分矩阵，列为：场景体系名称 / 主分类维度 / 核心测试场景 / 优化集样本数 / 保留集样本数 / 场景内比例 / 是否合规 / 缺口
- 标签覆盖矩阵，列为：覆盖标签 / 标签值 / 覆盖目的 / 覆盖样本数 / 缺口说明
- 样本量不足时，写明相关样本属于 `临时观察样本（未纳入正式拆分）`；不能把不同场景分别分配到不同集合后宣称拆分完成
- 多候选对比时，展示候选运行矩阵，列为：样本 ID / 核心测试场景 / 覆盖标签 / 候选 ID / 运行状态 / 输出路径 / 日志路径 / 是否复用 / 复用来源运行 ID / 是否计入质量结论
- 展示资源复用记录，列为：样本 ID / 候选 ID / 复用来源运行 ID / 复用产物路径 / 复用日志路径 / 复用原因 / 补跑范围 / 是否计入质量结论
- 对比评测报告中的场景适配建议必须基于同一场景体系和覆盖标签，不用混杂维度做胜负归因
- 评判方式与评判规则表，列为：指标 / 评判方式 / 评判规则 / 大语言模型及版本 / 评分细则路径或说明
- 数据记录表落盘位置

## 测试结果

> 本节只展示本轮真实测试得到的数据、现象和归因。

推荐小节：

1. `本轮执行概览`
2. `可唤起性与存在必要性检查`
3. `评测数据记录表`
4. `样本池构建与拆分结果`，必须包含评测场景体系表、场景级拆分矩阵、标签覆盖矩阵和覆盖缺口
5. `端到端测试`
6. `保留集样本测试`
7. `分歧样本明细`
8. `资源效率与结果复用`

多候选对比评测增加以下小节：

1. `候选运行覆盖`
2. `候选维度胜负表`
3. `场景适配建议`
4. `样本级成对比较`
5. `无 skill 基线对照`
6. `候选 trace 对比`

如出现 fallback / 兜底，增加记录表：

| 样本 ID | 候选/variant | 触发环节 | 原因分类 | 处理动作 | 是否计入质量结论 |
| --- | --- | --- | --- | --- | --- |

原因分类只能使用：网络问题、权限/凭证问题、外部服务限流或超时、运行环境/依赖问题、skill 逻辑问题、输入数据问题、评判器问题、用户中断、未知原因。

如复用已有结果，增加记录表：

| 样本 ID | 候选/variant | 复用来源运行 ID | 复用产物路径 | 复用日志路径 | 复用原因 | 是否计入质量结论 |
| --- | --- | --- | --- | --- | --- | --- |

如果某个小节没有实测数据，保留小节，但不能直接标为 `待补测` 了事。先说明已尝试或已判断的主动补测路径；只有当前工具、权限、数据或目标对象形态确实无法覆盖时，才标为 `待补测`，并说明阻塞原因和下一步应该怎么补测。

如果是文档型 / 工作流型 skill 的纯静态评测，使用这些结果小节：

1. `本轮执行概览`
2. `静态结构检查`
3. `可唤起性与存在必要性检查`
4. `发现的问题`
5. `待补测计划`，仅包含当前客观无法测量的项目

没有真实运行时，不要编造 endpoint、成本、耗时或样本质量数据。没有直接采集到 `token 成本`、`在线写入耗时` 或外部 trace UI 时，不要在状态总览和指标表中提这些关键词；只写已经拿到的 Codex 本地事件日志、CLI 日志、数据记录表和产物路径。

## Skill 优化方向

> 本节把评测结果反哺回 skill。每条优化方向都应能追溯到数据或失败样本。

使用表格：

| 优化项 | 来源证据 | 建议修改 | 验证方式 |
| --- | --- | --- | --- |

如果相关，补充可唤起性和无 skill 基线的后续动作。

如果优化项涉及修改 skill 的描述、规则、模板、工具链、评分器或参考文件，`验证方式` 必须写明补充样本或标签、重跑范围、对照口径和保留集抽测方式，用来确认修改后的行为确实符合预期。

多候选对比报告的优化方向要分成两类：对候选 skill 自身的修改建议，以及对评测资产的改进建议。不要把“另一个候选做得更好”直接写成唯一建议，要说明可借鉴的具体机制。

## 产物与复现位置

> 本节用于复现和后续接着做优化。

包含以下路径或链接：

- 原始日志
- 原始输出
- 全量数据集
- 评测场景体系表
- 数据集拆分：按核心测试场景保存的优化集 / 保留集矩阵
- 标签覆盖矩阵
- 评测设计确认记录
- 评测运行记录
- 测试样本记录
- 指标结果记录
- 本轮使用模型及版本记录
- 资源效率与结果复用记录
- 评分细则和模型评判提示词文件
- fallback / 兜底记录
- 评判输出
- 分歧样本
- 多候选对比的候选清单、候选运行矩阵、候选维度胜负表、场景适配表、候选 trace 记录
- 测试流程协同记录
- 生成的报告
- 优化方案
