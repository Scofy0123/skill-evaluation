# Skill 评估报告生成

使用这个 skill 来评估一个 Skill / Agent / 多环节 skill 链路，并产出一份简洁、基于证据、可复用的评估报告。报告既要能帮助当下做判断，也要能沉淀为后续复测资产。

## 硬性原则

- 不编造测试数据。没有真实测量的结果，必须标记为 `待补测` 或 `间接证据`，并说明下一步需要怎么补测。
- 必须主动穷尽当前可用工具补测。只有在当前权限、工具、数据或目标对象形态下客观无法测量的项目，才允许保留 `待补测` / `待验证`；不能因为没有现成脚本、需要额外清洗、需要写小工具、需要用规则评判、需要跑独立会话或需要复盘历史产物，就把项目留空。
- 禁止用改名掩盖未测状态。不能把 `待补测` 简单改成 `工程化缺口`、`不可直接测得`、`不适用` 来伪装完成；如果确实无法测量，必须说明已尝试的补测路径、阻塞原因、缺少的具体输入/权限/工具，以及下一步如何补齐。
- 设计指标前，必须先阅读目标 skill 文件和相关代码。
- 必须识别并评测目标 skill 的主功能产物。先回答“这个 skill 最核心应该产出什么”，例如 PRD、测试用例、代码补丁、评估报告、表格、图片、飞书文档、自动化运行结果等；正式报告中必须包含至少一个主功能产物的生成、复盘或明确阻塞说明。不能只做 metadata、触发、规则覆盖、静态结构检查，就把业务效果层写成已测。
- 如果当前环境可以安全构造样本，就必须至少跑通或模拟跑通 1 条端到端主功能样本，并把输入、生成产物、评判结果和产物路径写入测试样本记录与指标结果记录。只有当主功能依赖缺失权限、外部付费系统、不可替代真实数据或高风险副作用时，才允许不生成主产物，并必须把该项标为 `待补测`。
- 必须识别 AI 无法替代人工判断的评测项，并准备人工评测填写表。凡是涉及业务满意度、内容是否符合团队口径、PRD 详情质量是否让产品/研发/测试认可、视觉审美、策略取舍、金标争议、上线阈值等需要人类判断的内容，不能伪装成 AI 已完成评判。报告里必须插入可由人工填写的表格，至少包含样本 ID、产物链接/路径、人工评测维度、评分或通过状态、问题说明、修改建议、评审人、评审日期、最终结论；同时把该表保存为可复用的本地文件或在线表格。
- 固定使用三层架构：`运行框架层`、`能力链路层`、`业务效果层`。
- 指标表中的 `评判方式` 字段只能使用 `规则评判`、`模型评判`、`人工评判` 三类枚举；多种方式并用时只能组合这些枚举。脚本、Schema 校验、文件存在性、命令退出码、正则、阈值和确定性检查都归入 `规则评判`，不能写成旧称或其他自造方式。
- 每个指标都必须填写 `评判规则`：规则评判写清具体规则和脚本文件路径；模型评判写清模型提示词文件路径；人工评判写清人工评审步骤、通过标准、评分档位、需填写字段和复核责任。
- 报告和数据记录必须记录本轮实际使用的大语言模型及版本，至少区分任务执行模型、模型评判模型和无 skill 基线模型；无法取得精确版本时，记录可观测模型标识和来源。
- 任何需要打分或使用评分器的指标都必须有评分细则，至少包含分值档位、每档判定条件、证据要求、扣分规则和人工复核触发条件；模型评分器 prompt 必须引用同一套评分细则。
- 所有 fallback / 兜底情况都必须记录原因、触发环节、处理动作和是否计入质量结论。原因分类固定为：网络问题、权限/凭证问题、外部服务限流或超时、运行环境/依赖问题、skill 逻辑问题、输入数据问题、评判器问题、用户中断、未知原因。兜底产物默认只作为排障证据，不能计入正式质量结论，除非明确标为非独立证据。
- 评测过程中必须节约 token 和运行资源。先盘点已有有效证据，再补跑缺失证据单元；做存在必要性评测时，如果前序同样本、同输入、同版本、同模型的“有 skill”产物已经有效落盘，可复用为对照侧，只补跑独立无 skill 基线。复用必须记录来源运行 ID、产物路径、日志路径、模型及版本、复用原因和是否计入正式结论。
- 每次根据评测结果修改 skill 后，必须把“希望改善的点”转成补充样本或补充标签，重跑受影响的优化集，并抽测相关保留集或边界样本，验证修改是否真的带来预期改善且没有引入退化。
- 必须在关键节点和用户澄清需求，不能替用户“猜定”业务口径。至少在评测目标/指标、三层拆解、数据来源、数据集构建、人工评判口径、权限副作用和输出形态这些节点，列出已知信息、默认假设和需要用户确认的问题；用户未确认前，不要把合成样本、猜测指标或自行划分的数据集当作正式评测方案。
- 分层、评测目标、指标、数据方案和测试流程形成草案后，必须先给用户确认；用户确认前不要进入大规模测试、在线写入或最终报告生成。
- `请输出报告`、`请完成评测`、`使用 skill-evaluation 评测` 只表示最终目标，不等于跳过确认。除非用户明确写出 `跳过确认`、`无需确认直接执行` 或同等意思，否则必须在评测设计草案处暂停，等待用户回复。
- 评测设计确认必须作为一个显式门禁执行：向用户输出完整确认包，并以明确问题收尾，例如“请确认这套评测指标和流程是否通过；你回复‘确认，继续执行’后我再开始正式评测。”输出确认包后必须停止当前执行链，不能把确认请求夹在长报告中继续往下做。
- 只有用户明确表达确认（例如“确认”“可以”“继续执行”“按这个来”）才算通过门禁；用户只是补充数据、追问细节、讨论样本或要求“再看看”，都不算确认，必须先回答/修订确认包，再次请求确认。
- 如果用户提供在线数据源或在线文档，必须以当前在线内容为准。历史缓存、旧报告、临时文件只能作为辅助线索，不能替代在线回读；若使用缓存补充缺失字段，必须在数据记录和报告中标记为 `历史缓存/间接证据`，并说明为什么无法从在线源获取。
- 必须包含两个 Skill 专项目标：
  - `可唤起性`：需要使用这个 skill 时，agent 能否正确选中它。
  - `存在必要性`：相比无 skill / 通用模型临场处理，这个 skill 是否带来稳定收益。
- 必须设计并落地评测数据记录表。即使前期只是 CSV / JSONL / Markdown 表，也要保存运行脚本产生的样本、指标、日志、评判结果和产物路径。
- 评测场景必须来自同一套分类体系。先定义 `场景体系名称` 和 `主分类维度`，再列核心测试场景；同一组核心场景必须来自同一维度、同一粒度。语言、地域、技术主题、行业、风险点等横切因素应写成 `覆盖标签`，不能和核心场景混在一组里。
- 数据集拆分口径必须使用新版两类拆分：`优化集 / 保留集`。禁止擅自新增 `回归集` 或其他未确认的数据集类别。
- 数据集拆分必须是场景内拆分：先定义核心测试场景和覆盖标签，再在每个核心场景内部分别构建 `优化集 / 保留集`。不能把不同测试场景分别放进不同集合，也不能把不同维度的标签当作同一组核心场景。
- 如果某个场景样本量不足以同时覆盖 `优化集 / 保留集`，该场景的拆分状态必须写成 `场景级覆盖不足` 或 `临时观察样本（未纳入正式拆分）`，并在确认包和报告中说明限制、缺口和补样计划；不能只给全局比例，不能把全局 `50% / 50%` 写成场景级合规。
- 多候选 skill 对比评测必须使用共同数据集、共同评判口径和独立运行证据。每个候选对象都要单独固定来源版本、文件 SHA 或提交信息；每条主功能样本原则上要形成可追溯的 `样本 × 候选 skill × 无 skill 基线` 证据矩阵。已验证有效的矩阵单元可以复用，缺失或失效单元再补跑。失败、中断和兜底产物必须分开记录；非独立运行或当前会话补写产物不能计入候选 skill 的质量结论。
- 多候选对比报告必须明确回答“哪个更好用、适合什么场景、差距在哪里”。覆盖不足时允许给阶段性结论，但必须说明已覆盖场景、未覆盖场景和不能下最终结论的原因；不能用单一总分掩盖场景差异。
- `存在必要性` 只有在对同一批样本执行了独立、受控的无 skill 基线或等价隔离对照后，才允许写成 `通过` 或 `部分通过`。无 skill 基线可以和前序已验证有效的“有 skill”结果做成对比较，不要求重复跑有 skill 侧。若只是静态分析、历史产物复盘、弱隔离 prompt、同上下文人工估计或主观比较，只能标为 `间接证据 + 待独立基线`，不能写成已验证稳定收益。
- 除非用户明确要求并接受取舍，否则不要把所有指标加权合成一个总分。
- 如果写入飞书或其他在线文档，优先做局部章节更新并尽量保留评论；避免整篇覆盖。
- 如果需要把评测结果文件上传到飞书云盘，默认逐个文件单独上传，禁止把多个结果文件打成 zip 后只上传压缩包，除非用户明确要求压缩包。面向用户展示或交付的附件文件名必须使用中文名称，避免使用 `metric_results.json`、`sample_results.csv` 这类英文技术文件名作为最终飞书文件名。
- 如果评测通过 Codex / Codex Desktop / Codex CLI 执行，必须优先采集可用的 Codex 本地事件日志，而不是笼统写“缺少 trace”。可用事件至少包括：`session_meta`（会话 ID、cwd、模型、来源）、`turn_context`（turn ID、权限、sandbox、模型设置）、`response_item.function_call`（工具调用名称与参数）、`response_item.function_call_output`（工具输出、退出状态或错误）、`event_msg.token_count`（如存在则记录 token usage）、`event_msg.task_complete`（任务完成时间与耗时）。常见来源包括 `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl`、`~/.codex/archived_sessions/rollout-*.jsonl`、`~/.codex/logs_2.sqlite`，以及 `codex exec --json` 产生的 JSONL 事件流。
- 不要把拿不到的观测项写成报告里的醒目缺口。`token 成本`、`总成本`、`在线写入耗时`、`真实 trace UI` 等只有在本轮确实采集到对应数据时才写入报告；如果没有采集到，就从状态总览和指标表中移除这些关键词，改写为已经采集到的证据，例如“Codex 本地事件日志”“工具调用输出”“CLI 执行日志”“数据记录表”。若必须说明未测副作用，只写具体事实，例如“未执行真实写入，因此未评估写入链路”，不要泛化成“缺少在线写入耗时”。
- 最终报告只能呈现本轮事实、测量结果、证据路径、限制和下一步建议，禁止把 skill 内部工作法、写作规范、评测口径或给评估者的操作指令原样写入正文。凡是来自本 skill 的方法论规则，必须先转写成“本轮实际做了什么 / 采集到什么 / 没有执行什么 / 因此结论是什么”的事实陈述；不要在最终报告里出现“如果使用...应...”“报告后续只写...”“不要...而要...”这类指导性句式，除非它们位于明确标注为“后续改进建议”的章节且面向被评测对象。

## 工作流程

以下步骤是标准 7 步工作法的执行展开：`建立数据记录表` 属于“制定测试流程”和“跑测试并记录数据”的必要动作，`撰写报告` 属于“分析并回流”后的结构化输出动作，不能省略。

1. **收集上下文**（参考 [evaluation-runbook.md](references/evaluation-runbook.md) 的 `RUN-01`）
   - 阅读目标 `SKILL.md`、引用文件、脚本、日志、trace / metrics、测试产物和已有报告。
   - 如果当前评测在 Codex 中执行，采集或记录本轮 Codex 事件证据：会话 JSONL 路径、关键事件类型、工具调用/输出、token_count（如存在）、task_complete（如存在）。如果是 `codex exec`，优先使用 `--json` 保存事件流。
   - 如果用户提供了工作法文档或参考样例，先阅读它们，并把它们作为报告结构基线。
   - 如果目标包含多个相互依赖的处理环节，先识别链路顺序、依赖关系和边界。

2. **理解评测对象**（参考 [evaluation-runbook.md](references/evaluation-runbook.md) 的 `RUN-02`）
   - 总结使用场景、成功标准、skill 做什么、输入是什么、输出是什么、依赖什么、边界在哪里。
   - 如果是多候选对比评测，先建立候选对象清单：候选 ID、来源路径或 URL、版本 / SHA、核心职责、主功能产物、已知边界；再定义共同任务口径，避免把不同定位的 skill 硬拉成同一指标。
   - 明确主功能产物：写出目标 skill 最核心的交付物、最低可接受样例、产物保存格式，以及如何判断产物是否合格。
   - 标出需要人工确认的业务口径、金标样本、争议样本和上线阈值。
   - 如果使用场景、成功标准、主功能产物、候选边界或人工业务口径不清楚，先把不确定点列成澄清问题，不要直接进入指标设计。
   - 区分它真正负责的事情，以及发布、通知、存储、UI 等相邻模块。
   - 如果 skill 的链路较复杂，把额外观察点纳入三层内部，不新增层级。

3. **建立三层模型**（参考 [evaluation-runbook.md](references/evaluation-runbook.md) 的 `RUN-03`）
   - `运行框架层`：环境、凭证、权限、API、日志、可用事件、输出文件、路由/唤起行为；成本、耗时等仅在本轮实际采集到时纳入。
   - `能力链路层`：内部能力链路、数据转换、Schema 完整性、工具编排、状态交接。
   - `业务效果层`：最终任务质量、相关性、准确性、用户价值、误判/漏判、业务取舍。
   - 三层拆解形成草案后，在确认包中请用户确认是否符合真实职责边界；如果用户指出层级或边界不对，先修订后再继续。

4. **定义目标和指标**（参考 [evaluation-runbook.md](references/evaluation-runbook.md) 的 `RUN-04`）
   - 把每一层转成可观测指标和计算规则。
   - 指标设计前先澄清本轮最重要的业务目标、主要风险、样本优先级、人工评判口径和不希望评测的范围；用户没有提供时，在确认包中写成待确认假设。
   - 必须加入 `主功能产物质量` 指标，例如是否生成目标产物、模板/Schema 是否完整、核心字段是否覆盖、产物是否满足输入目标、是否保存可复现路径。
   - 必须加入 `人工评测项` 识别：列出哪些质量维度可由规则或模型先评，哪些必须由人类最终判断；对人工项要设计填写表，而不是把它们合并进 AI 自评分。
   - 加入 `可唤起性` 指标，例如目标提示词命中率、负例不误触率、相似 skill 干扰正确率。
   - 加入 `存在必要性` 指标，对比有 skill 和无 skill 基线在质量、稳定性、可复现性、资产沉淀上的差异；如果有 skill 侧已有有效运行产物，优先复用并记录复用证据。
   - 多候选对比评测还要加入 `候选间胜负关系`、`场景适配性`、`trace 可解释性`、`运行失败归因` 指标；使用维度胜负表和样本级成对比较，不默认合成总分。
   - 每个指标都要写清数据来源、计算规则、评判方式、评判规则、大语言模型及版本、评分细则路径或说明，以及解释口径。

5. **确认评测设计**（参考 [evaluation-runbook.md](references/evaluation-runbook.md) 的 `RUN-05`）
   - 输出一版评测设计草案，至少包含：三层拆解、评测目标、指标表、数据来源、数据集拆分、评判方式与评判规则、数据记录表结构和测试流程。
   - 在确认包中提醒用户补充更关心的业务目标、评测方向、重点场景、风险点和人工判断口径；如果用户不补充，按当前证据和通用评测目标继续。
   - 确认包中必须单列 `需要用户澄清或确认的关键问题`：目标/指标是否准确、三层拆解是否符合真实职责、用户能否提供数据集、是否允许构造合成样本、样本规模和拆分比例是否接受、人工评判由谁填写、是否允许在线写入或调用外部服务。
   - 评测场景必须有体系：先写清 `场景体系名称`、`主分类维度`、核心测试场景和覆盖标签。需要具体例子或反例时，读取 [scenario-design-examples.md](references/scenario-design-examples.md)。
   - 请用户确认是否符合预期；如用户提出调整，先迭代草案。
   - 这一步是阻塞点。输出草案后必须停止当前执行链，不能继续导出数据、批量运行、落盘结果、生成最终报告或写入飞书。
   - 用户确认前，只做低风险的资料读取、静态检查和小范围可逆验证。
   - 确认包必须使用下面的最小结构，避免漏问指标：
     1. `评测对象与证据范围`
     2. `三层拆解`
     3. `评测目标与指标表`，每个指标写清所属层级、计算规则、数据来源、评判方式、评判规则、大语言模型及版本、评分细则路径或说明、复用证据
     4. `数据方案与数据集拆分`，必须包含场景体系名称、主分类维度、核心场景清单、覆盖标签、每个场景内的 `优化集 / 保留集` 样本数量、比例合规结果和缺口说明；多候选对比时还要包含候选运行矩阵和无 skill 基线方案
     5. `评判方式与评判规则`
     6. `主功能产物评测方案`，说明至少一个端到端产物样本如何生成、保存和评判；如果暂不能生成，说明阻塞原因和补测路径
     7. `人工评测项与填写表方案`，说明哪些内容必须人工评估，人工需要填写哪些字段，表格保存在哪里
     8. `测试流程与数据记录表`，必须说明哪些结果会复用、哪些缺失单元需要补跑
     9. `副作用与权限边界`
     10. `需要用户澄清或确认的关键问题`
   - 确认包最后必须单独提出确认问题，不能只写“我将继续”或“接下来会执行”。

6. **设计测试流程**（参考 [evaluation-runbook.md](references/evaluation-runbook.md) 的 `RUN-06`）
   - 说明测试数据集怎么构建：数据从哪里来、怎么清洗、怎么标注样本属性、如何准备 reference output 或人工金标。
   - 先问用户是否能提供真实数据集、历史样本、生产/业务样例、金标或团队认可的参考产物；用户无法提供时，再提出合成样本或历史产物复盘方案，并在确认包中标明数据来源和限制。
   - 数据集构建方案不能自行默认为“评测执行者构造”。如果需要构造合成样本，必须写清样本量、场景覆盖、生成原则、人工确认点和是否进入正式结论。
   - 说明数据怎么筛选、怎么拆分、用什么评判方式、如何执行、失败如何回流。
   - 先设计评测场景体系：定义 `场景体系名称`、`主分类维度`、核心测试场景、场景定义、预期观察点和覆盖标签。核心场景必须同维度、同粒度；覆盖标签不能替代核心场景。
   - 数据集拆分必须使用新版两类类别：`优化集 / 保留集`，不新增第三类；拆分前先定义核心测试场景，并在每个场景内分别分配样本。
   - 拆分比例必须结合 skill 阶段选择，并在每个核心场景内部校验比例区间。样本量不足时，不要为了保留两类名称而把不同场景硬分到不同集合；应把相关样本标为 `临时观察样本（未纳入正式拆分）`，并写清“当前只是最小观察集，不足以形成场景级优化/保留拆分”。
   - 多候选对比时，每条样本使用同一输入分别形成各候选 skill 和无 skill 基线的证据单元；运行上下文、prompt、输出保存和评判器尽量保持一致，差异只来自被测候选对象。已有有效证据单元可复用，缺失或失效单元再补跑。
   - 按场景选择评判方式：确定性产物、脚本、Schema、文件、退出码、正则和阈值检查用 `规则评判`；语义初筛用 `模型评判`；金标、业务取舍和团队口径用 `人工评判`。
   - 在测试流程内部判断是否需要多 agent / sub-agent 协作：当评测对象需要拆成多个相对独立的观察面，且需要隔离上下文、独立产证时，可以设计协同编排；只有在用户明确要求或当前环境允许时才实际启用。

7. **建立数据记录表**（参考 [evaluation-runbook.md](references/evaluation-runbook.md) 的 `RUN-07`）
   - 在执行前定义评测结果数据表结构，并把存档位置写入计划。
   - 至少保存三类数据：评测运行记录（运行 ID、候选 ID 或 variant、执行命令、环境、数据版本、任务执行模型及版本、模型标识来源、已采集到的耗时或事件时间、输出目录、Codex 会话 JSONL 或 CLI JSONL 路径、是否补跑、复用来源运行 ID）、测试样本记录（样本 ID、候选 ID 或 variant、场景体系名称、主分类维度、核心测试场景、覆盖标签、场景内数据集拆分、正式拆分状态、输入、预期输出、实际输出、错误、产物路径、fallback 原因、fallback 是否计入结论、是否复用、复用产物路径）、指标结果记录（候选 ID 或 variant、所属层级、指标名、评判方式、评判规则、模型及版本、评分细则路径或说明、分数或标签、通过状态、解释、复核状态、复用证据说明）。
   - 如果存在人工评测项，必须额外保存人工评测填写表。表格可先为空，但字段必须完整，且每一行要绑定具体样本和产物路径/链接，方便评审人直接填写。
   - 优先使用项目内 `eval_runs/`、`datasets/`、`results/` 等目录保存 CSV / JSONL / SQLite；如果用户指定 LangSmith、飞书多维表格或数据库，则按指定系统落地。
   - 若后续需要上传到飞书云盘，本地可保留英文技术文件名方便脚本处理，但上传时必须映射为中文文件名，并逐文件上传。

8. **运行或复盘测试**（参考 [evaluation-runbook.md](references/evaluation-runbook.md) 的 `RUN-08`）
   - 在安全且已授权的情况下，优先本地实际运行目标 skill；已有有效证据时先复用，再补跑缺失的无 skill 基线或候选结果。
   - 如果目标没有现成执行入口，先判断能否通过以下方式主动补测：写小型评测脚本、用规则评判、做静态结构校验、复盘历史产物、用飞书/CLI/API 拉取证据、构造负例和边界样本、启动独立会话做无 skill 基线、或把目标文档型流程转成可评判的样本记录。
   - 对主功能产物单独执行一次“产物存在性检查”：报告必须能指向至少一个生成产物或历史产物路径，并有对应的质量评判记录。若没有产物路径，必须在测试结果中明确写“未生成主功能产物”，不能只展示规则覆盖率。
   - 对每个原计划标为 `待补测` / `待验证` 的项目，必须先做一次“可主动补测性”判断；能测就测，不能测才保留状态。
   - 记录精确命令、退出码、输出文件、可用的事件日志和样本级结果，并写入评测数据表。只有确实采集到了成本或耗时，才记录成本或耗时；没有采集到时不要在报告中保留空指标。
   - 如果只使用历史日志，必须明确证据来源，不要把复盘分析写成“本轮新跑”。
   - 如果目标是无可执行入口的文档/工作流型 skill，使用静态评测和复盘评测模式：读取文件、校验 metadata、检查流程和模板覆盖、复盘历史输出、构造规则评判和基线对照；只有这些方法仍无法覆盖的真实行为，才标为 `待补测`。

9. **分析结果**（参考 [evaluation-runbook.md](references/evaluation-runbook.md) 的 `RUN-09`）
   - 先展示测量数据，再解释含义。
   - 当规则、模型评判或人工标签不一致时，列出分歧样本。
   - 对人工评测项保持状态透明：未填写前标为 `人工待评`，不能写成 `通过`；已填写后再按人工结果更新结论。
   - 把失败点同时沉淀为 eval 资产和 skill 优化项。

10. **撰写报告**（参考 [evaluation-runbook.md](references/evaluation-runbook.md) 的 `RUN-09`）
   - 使用 [报告模板](references/report-template.md) 的结构。
   - 面向不懂评测的读者，每个主要章节开头用简短引用说明本节在做什么。
   - 在报告最前面加入一张状态总览表，列出运行框架层和能力链路层的关键检查项，用 `✅ / ❌ / 待补测` 展示是否满足。
   - 指标、步骤、数据集拆分、样本明细、优化项优先用表格。
   - 发布或写入在线文档前，必须做“报告成品口吻自检”：搜索并改写指导性元话语，例如 `如果使用`、`报告后续`、`应优先`、`不要...写成`、`只写`、`必须先`、`建议使用`。保留真正属于被评测 skill 优化建议的内容；其他内部评测方法说明必须改成已执行事实或移出最终报告。
   - 只有当图能显著解释架构或流程时才加图；图里的逻辑必须和正文一致。
   - 多候选对比报告使用同一模板，但要在摘要、测试结果和优化方向中补充候选间对照表、场景适配建议、样本级成对比较、每个候选的 trace 图或 trace 表。

## 输出要求

最终报告必须包含：

- 评测对象和证据范围
- 三层拆解
- 评测目标与指标，包括 `可唤起性` 和 `存在必要性`
- 评测设计确认记录
- 测试流程与每一步输出物
- 测试流程中的 multi-agent / sub-agent 协同编排判断
- 数据准备与数据集拆分依据
- 评测场景体系表，包括场景体系名称、主分类维度、核心场景、场景定义、预期观察点、覆盖标签和场景级合规状态
- 每个核心测试场景在已确认数据集类别下的覆盖情况与比例合规结果；必须使用场景级 `优化集 / 保留集` 矩阵，不得只展示全局比例；如果无法做到场景级覆盖，必须说明原因和补样计划
- 标签覆盖矩阵。标签只能解释横切覆盖情况，不能替代核心测试场景或场景内拆分
- 评测数据记录表结构和真实落盘位置
- 评判方式与评判规则，包括固定枚举、脚本/提示词/人工规则路径、模型及版本、评分细则
- 资源效率与结果复用记录，包括复用来源运行 ID、样本 ID、候选 ID、产物路径、日志路径、复用原因、补跑范围和是否计入质量结论
- 主功能产物评测结果，包括输入样本、生成产物路径、质量评判结果；如果没有生成，必须说明为什么不能生成
- 人工评测填写表。只要存在 AI 无法替代人工判断的维度，就必须在报告正文插入可填写表格，并保存可复用文件；未填写时状态标为 `人工待评`
- 多候选对比评测结果，包括候选清单、共同数据集、候选运行矩阵、候选维度胜负表、场景适配建议、样本级成对比较、无 skill 基线对照和每个候选的 trace 记录
- fallback / 兜底记录，包括原因分类、触发环节、处理动作和是否计入质量结论
- 真实测得的结果
- 失败点 / 分歧样本分析
- Skill 优化方向
- 可复现产物和参考来源

如果是静态评测，也必须保留完整报告结构。已测的静态发现标为 `已测`；未运行的真实行为不能直接标为 `待补测`，必须先说明为什么无法通过脚本、规则评判、历史产物复盘、独立会话基线或其他当前可用工具主动测量。确认客观无法测量后，才标为 `待补测`，并写清已尝试路径、阻塞原因和下一步需要补什么。

如果用户要求新建在线文档，创建或更新文档后返回链接。如果用户要求本地产物，把 Markdown 写到指定仓库或工作目录。
